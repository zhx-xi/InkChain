# InkOS 安全策略

## 概述

InkOS 将用户数据安全放在首位。作为一款本地优先的 AI 写作工具，我们致力于确保软件的安全性和透明度。本文件说明了我们的安全实践和漏洞报告流程。

## 数据处理与存储

### 数据位置
- **全部数据本地存储**：所有创作内容、配置、API 密钥均存储在您的本地设备上
- **无云端数据库**：InkOS 不连接任何集中式数据库或云端存储
- **无数据传输**：InkOS 不会将您的数据上传到任何外部服务器（除非您主动使用第三方 AI API）

### API 密钥安全
- API 密钥存储在 `.inkos/secrets.json`（项目本地）或环境变量中
- ⚠️ **当前限制**：API 密钥以明文存储。我们正在开发系统密钥链集成以增强安全性
- **建议**：
  - 不要分享 `.inkos/secrets.json` 文件
  - 不要在公开的代码仓库中提交该文件（已通过 `.gitignore` 包含）
  - 定期轮换 API 密钥
  - 为 API 密钥设置合理的使用限制和预算告警

## 安全最佳实践（用户指南）

### 保护您的创作内容
```bash
# 定期备份您的书籍项目
cp -r ~/your-project/books ~/backup/books-$(date +%Y%m%d)

# 使用加密备份（推荐）
tar czf - ~/your-project | openssl enc -aes-256-cbc -out backup.tar.gz.enc
```

### API 密钥管理
- 为每个服务设置单独的使用限额
- 启用 LLM 服务商的计费告警
- 不要重用跨项目的 API 密钥
- 如果怀疑密钥泄露，立即在服务商平台撤销

### 网络安全
- 如果配置了 Webhook 通知，使用 HTTPS 端点
- Webhook 请求使用 HMAC-SHA256 签名（如果配置了 `secret`）
- 本地 Studio 服务（端口 4567）仅监听 localhost，不暴露到公网

## 已知安全考量

| 项目 | 状态 | 说明 |
|------|------|------|
| API Key 存储 | ⚠️ 明文存储 | 计划支持操作系统密钥链（Windows Credential Manager / macOS Keychain / Linux Secret Service） |
| 本地服务 CORS | ℹ️ 默认全开 | 本地工具使用，本地网络范围内可接受 |
| 速率限制 | ℹ️ 未实施 | 本地单用户工具，暂不需要 |
| 路径遍历防护 | ✅ 已实施 | BookId 和文件路径经过安全校验 |
| SQL 注入 | ✅ 不适用 | 使用参数化查询（`node:sqlite`） |

## 报告安全漏洞

我们非常重视安全问题。如果您发现了安全漏洞，请遵循以下流程：

### 报告流程

1. **请勿公开披露**：请勿在 GitHub Issues、讨论区或其他公开渠道披露安全漏洞
2. **加密沟通**：如果需要发送敏感信息，请使用 GPG 加密
3. **发送报告**：通过 GitHub Security Advisories 或邮件联系维护者

### 报告内容

请包含以下信息以帮助我们快速响应：

- 漏洞的详细描述
- 复现步骤
- 受影响的版本
- 可能的缓解措施（如果您有建议）
- 您的联系方式（以便后续沟通）

### 我们承诺

- **响应时间**：在 5 个工作日内确认收到您的报告
- **状态更新**：在漏洞修复过程中定期（不超过每 14 天）更新进展
- **修复时间表**：对于严重漏洞，在确认后 30 天内发布修复
- **致谢**：在修复发布后，如您同意，我们将在 Release Notes 中致谢
- **安全披露**：修复发布后，我们将发布安全公告说明漏洞及其影响范围

## 依赖安全

InkOS 使用多个开源依赖。我们建议：

- 使用 `pnpm audit` 定期审查依赖安全
- 关注关键依赖的安全公告
- 及时更新依赖至最新安全版本
